# 🏥 Hospital Honeypot — Terraform Deployment Guide

Automated AWS infrastructure deployment for a hybrid hospital honeypot environment. This Terraform project provisions a full simulated hospital network (VMs, containers, VPC, subnets, security groups, DNS) designed to attract and log attacker behavior via an ELK stack.

---

## 📁 File Overview

| File | Purpose |
|---|---|
| `main.tf` | AWS provider config |
| `variables.tf` | AMI ID variables (dynamically resolved) |
| `vpc.tf` | VPC, subnets, internet gateway, route tables |
| `security_groups.tf` | Firewall rules for Brain and Clinical zones |
| `instances.tf` | All EC2 instances + user_data bootstrap scripts |
| `keys.tf` | SSH key pair resource |
| `dns.tf` | Route 53 private hosted zone + DNS records |
| `outputs.tf` | Prints Brain public IP and Kibana URL after apply |
| `data.tf` | Dynamic AMI lookups (Ubuntu 22.04, Windows 2012 R2) |

## 🚀 Deployment Steps

### Step 1 — Generate the SSH Key

This **must be done before `terraform apply`**. Terraform will look for this key to register it as a key pair in AWS.

```bash
ssh-keygen -t rsa -b 4096 -f ~/.ssh/honeypot_key
```

This creates two files:
- `~/.ssh/honeypot_key` — your private key (never share this)
- `~/.ssh/honeypot_key.pub` — the public key that gets uploaded to AWS

### Step 2 — Initialize Terraform

Download the required AWS provider plugins:

```bash
terraform init
```

### Step 3 — Preview the Plan

Review everything Terraform intends to create before committing:

```bash
terraform plan
```

Check that the AMI IDs resolve correctly — they are dynamically fetched, so the output will confirm what was selected. The two relevant outputs to watch are:
- `ubuntu_ami_used`
- `windows_ami_used`

### Step 4 — Apply / Deploy

```bash
terraform apply
```

Type `yes` when prompted. Terraform will output the Brain's public IP and Kibana URL when finished:

```
brain_public_ip = "x.x.x.x"
elk_kibana_url  = "http://x.x.x.x:5601"
```

### Step 5 — Wait for Bootstrap to Complete (~5 minutes)

All instances run `user_data` scripts on first boot to install and configure software automatically. Give it about 5 minutes for:

- The Brain: Docker, Docker Compose, Elasticsearch, Logstash, Kibana
- Imaging Server: Java, DCM4CHE (DICOM), Filebeat
- IoT Gateway: Docker, Conpot (medical template), Filebeat
- Windows Workstation: Winlogbeat download, config, and service install

### Step 6 — Validate Deployment

SSH into The Brain and confirm ELK is running:

```bash
ssh -i ~/.ssh/honeypot_key ubuntu@<brain_public_ip>
docker ps
```

You should see containers for `elasticsearch`, `logstash`, and `kibana` all with status `Up`.

Then open Kibana in your browser:

```
http://<brain_public_ip>:5601
```

If Kibana loads, the stack is healthy. ✅

## 🔒 Post-Validation Hardening (DO THIS AFTER KIBANA IS CONFIRMED)

Once the environment is validated and working, the temporary bootstrap access must be removed to comply with AWS ToS and isolate the honeypot. There are **two options**:

---

### Option A — Terraform (Recommended)

Edit `security_groups.tf` and `vpc.tf`, then re-apply. Terraform will update only the affected resources without touching your running instances.

**In `security_groups.tf`:**

1. **Brain SG** — Replace the temporary egress block with the locked-down version:

```hcl
# REMOVE this temporary block:
egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
}

# REPLACE with this:
egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["127.0.0.1/32"] # Block all outbound
}
```

2. **Clinical SG** — Same swap, replace the open egress with the loopback block.

**In `vpc.tf`:**

3. **Remove the Clinical Zone's public route table association** — delete or comment out this block:

```hcl
# REMOVE after deployment validation:
resource "aws_route_table_association" "clinical_rta" {
  subnet_id      = aws_subnet.clinical_zone.id
  route_table_id = aws_route_table.public_rt.id
}
```

**In `instances.tf`:**

4. **Remove public IPs from Clinical Zone instances** — set the following on `win_workstation`, `imaging_server`, and `iot_gateway`:

```hcl
associate_public_ip_address = false   # Remove public IP after bootstrap
```

Then apply:

```bash
terraform plan   # Confirm only security group + association changes
terraform apply
```

---

### Option B — AWS Console (Manual, No Terraform Required)

If you'd prefer not to re-run Terraform:

1. Go to **EC2 → Security Groups** in the AWS Console
2. Find `brain-sg` → **Edit outbound rules** → delete the `0.0.0.0/0` rule
3. Find `clinical-sg` → **Edit outbound rules** → delete the `0.0.0.0/0` rule
4. Go to **VPC → Route Tables** → find `Hospital-Public-RT` → **Subnet Associations** → disassociate the `Clinical-Zone` subnet

> ⚠️ If you go the manual route, note that your `terraform.tfstate` will be out of sync with actual AWS state. Run `terraform plan` afterward to review the drift.

## ⚠️ Known Limitations & TODOs

- **No `terraform.tfvars`** — AMI IDs and region are resolved dynamically via `data.tf`, but the region is still only configurable via `variables.tf`. Consider adding a `.tfvars` file for multi-region support.
- **Docker Compose in `user_data`** — Inline heredoc YAML is fragile. If the Brain fails to start ELK, consider migrating the `docker-compose.yml` to an S3 bucket and pulling it down in `user_data` with `aws s3 cp`.
- **Windows PowerShell script** — The Winlogbeat install script in `instances.tf` was written for PowerShell 3.0/4.0 compatibility and should be reviewed and tested carefully.
- **No state backend** — `terraform.tfstate` is stored locally. For team use, configure a remote backend (S3 + DynamoDB lock).

## 🗑️ Teardown

To destroy all resources and avoid ongoing AWS charges:

```bash
terraform destroy
```

Type `yes` when prompted. This will remove all EC2 instances, the VPC, subnets, security groups, Route 53 zone, and the key pair.
