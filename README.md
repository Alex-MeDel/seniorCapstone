# 🏥 Hospital Honeypot — Senior Capstone Project
#### Group 4

A cloud-based honeypot environment that simulates a vulnerable hospital network to observe, capture, and analyze cyberattack patterns targeting healthcare infrastructure — deployed entirely on AWS using Terraform and an ELK stack.

> ⚠️ **Disclaimer:** This environment is strictly isolated within a private AWS VPC with no outbound internet access post-deployment. It is designed for academic research only and is fully compliant with the AWS Acceptable Use Policy. No real patient data is used at any point.

---

## 📌 Project Overview

The environment deploys a realistic dual-zone hospital network as a passive honeypot. The **Clinical Zone** contains intentionally vulnerable legacy medical assets acting as decoys. The **Brain Zone** runs a containerized ELK stack to aggregate, process, and visualize all captured logs.

The goal is to study attack behaviors targeting healthcare-specific protocols — DICOM imaging servers, Modbus IoT gateways, and legacy Windows workstations — without performing any active attacks or violating AWS policies.

Administrative access to Kibana and all instances is restricted to an AWS Client VPN. All bootstrap scripts and config files are stored in a randomly-suffixed S3 bucket and pulled down at instance startup via IAM instance roles — no hardcoded credentials anywhere in the codebase.

## 🏗️ Architecture

The environment runs inside a single AWS VPC (`10.0.0.0/16`) split across two subnets:

```
AWS VPC: 10.0.0.0/16
│
├── 🏥 Clinical Zone (10.0.1.0/24)  ← Vulnerable honeypot decoys
│   ├── Medical Workstation   — Windows Server 2012 R2 + Winlogbeat   (dynamic IP)
│   ├── Imaging Server (PACS) — 10.0.1.20  Ubuntu 22.04 + DCM4CHE (DICOM port 104)
│   └── IoT Gateway           — 10.0.1.30  Ubuntu 22.04 + Conpot (HTTP/80, Modbus/502)
│
└── 🧠 Brain Zone (10.0.2.0/24)     ← Management & logging
    └── The Brain / ELK       — 10.0.2.10  Ubuntu 22.04 + ELK Stack via Docker Compose
```

**Internal DNS** (`hospital.internal` — Route 53 Private Hosted Zone):

| Hostname | Resolves To |
|---|---|
| `pacs.hospital.internal` | 10.0.1.20 |
| `iot-gateway.hospital.internal` | 10.0.1.30 |
| `medical-workstation.hospital.internal` | Windows private IP (dynamic) |

**Traffic Flow:**
1. Attacker (or validation script) probes Clinical Zone assets
2. Filebeat / Winlogbeat ships logs to The Brain on port 5044
3. Logstash ingests and forwards to Elasticsearch (port 9200)
4. Kibana visualizes on port 5601 (accessible via Client VPN only)

## 🚀 Deployment

### Step 1 — Generate the SSH Key

**Do this before `terraform apply`.** Terraform registers this key pair in AWS; it must exist on disk first.

```bash
ssh-keygen -t rsa -b 4096 -f ~/.ssh/honeypot_key
```

This creates:
- `~/.ssh/honeypot_key` — private key (never share or commit)
- `~/.ssh/honeypot_key.pub` — uploaded to AWS by Terraform

### Step 2 — Initialize Terraform

```bash
terraform init
```

### Step 3 — Preview the Plan

```bash
terraform plan
```

Check that `ubuntu_ami_used` and `windows_ami_used` appear in the planned outputs — these confirm the AMIs resolved correctly before you commit.

### Step 4 — Deploy

```bash
terraform apply
```

Type `yes` when prompted. On completion, Terraform will print:

```
brain_public_ip  = "x.x.x.x"
elk_kibana_url   = "http://x.x.x.x:5601"
ubuntu_ami_used  = "ami-xxxxxxxxxxxxxxxxx"
windows_ami_used = "ami-xxxxxxxxxxxxxxxxx"
```

### Step 5 — Wait ~5 Minutes for Bootstrap

All instances pull their bootstrap scripts from S3 via IAM instance role and run them on first boot. Expected bootstrap time per instance:

- **The Brain:** Docker, Docker Compose, pulls `docker-compose.yml` and `logstash.conf` from S3, launches Elasticsearch 8.10.2 / Logstash 8.10.2 / Kibana 8.10.2
- **Imaging Server:** Java, DCM4CHE 5.31.0 (StoreSCP on port 104), Filebeat 8.x
- **IoT Gateway:** Docker, Conpot with medical template (HTTP/80, Modbus/502), Filebeat 8.x
- **Windows Workstation:** Winlogbeat 8.10.2 downloaded via PowerShell from S3, installed as a Windows service

### Step 6 — Confirm ELK is Running

SSH into The Brain and verify the containers are up:

```bash
ssh -i ~/.ssh/honeypot_key ubuntu@<brain_public_ip>
docker ps
```

You should see three containers: `elasticsearch`, `logstash`, and `kibana`. Then open Kibana:

```
http://<brain_public_ip>:5601
```

If Kibana loads, the stack is healthy. ✅

You can also verify Elasticsearch received data with:

```bash
curl http://10.0.2.10:9200/_cat/indices
```

## 🧪 Validation

Once the environment is up and Kibana is accessible, run the validation script **from The Brain instance** to generate synthetic traffic across all Clinical Zone assets and confirm the full logging pipeline end-to-end.

```bash
python3 validationScript.py
```

**What the script does:**

Performs passive TCP "knocks" — Python `socket.connect_ex()` calls that initiate TCP handshakes without sending any exploit payload. It tests all three honeypot hosts across four protocol-relevant ports:

| Port | Protocol | Relevant Asset |
|---|---|---|
| 80 | HTTP | IoT Gateway (Conpot) |
| 104 | DICOM | Imaging Server (DCM4CHE) |
| 445 | SMB | Windows Workstation (Winlogbeat) |
| 502 | Modbus | IoT Gateway (Conpot) |

> **Note:** The script knocks all four ports against all three hosts. Not every port will be open on every host — `[KNOCK]` in the output is expected and still generates a loggable network event that Filebeat or Winlogbeat will capture and forward to The Brain.

**How to confirm success:**

1. Run the script from The Brain
2. Open Kibana at `http://10.0.2.10:5601` (via VPN)
3. Confirm log entries are appearing from Clinical Zone hosts
4. Verify connection attempts on the expected ports are visible in the index

A successful run proves the full pipeline — **Clinical Zone → Beats → Logstash → Elasticsearch → Kibana** — is operational without any active attack or AWS policy violation.

## 🔒 Post-Validation Hardening

Once Kibana is confirmed working, the temporary bootstrap internet access must be removed. Two options:

### Option A — Terraform (Recommended)

Edit the files below, then run `terraform apply`. Terraform will update only the affected security groups and route table associations without touching running instances or their data.

**1. `security_groups.tf` — Replace the open egress on both `brain-sg` and `clinical-sg`:**

```hcl
# REMOVE:
egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
}

# REPLACE WITH (on brain-sg):
egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["127.0.0.1/32"]
}

# REPLACE WITH (on clinical-sg) — preserve DNS + log shipping:
egress {
    from_port   = 0
    to_port     = 65535
    protocol    = "tcp"
    cidr_blocks = ["10.0.2.0/24"]   # Log shipping to Brain
}
egress {
    from_port   = 53
    to_port     = 53
    protocol    = "udp"
    cidr_blocks = ["10.0.0.0/16"]   # Internal DNS resolution
}
```

**2. `vpc.tf` — Remove the Clinical Zone's public route association:**

```hcl
# DELETE or comment out:
resource "aws_route_table_association" "clinical_rta" {
  subnet_id      = aws_subnet.clinical_zone.id
  route_table_id = aws_route_table.public_rt.id
}
```

**3. `instances.tf` — Remove public IPs from Clinical Zone instances:**

Set `associate_public_ip_address = false` on `win_workstation`, `imaging_server`, and `iot_gateway`.

Then apply:

```bash
terraform plan   # Confirm only security group + association changes
terraform apply
```

### Option B — AWS Console

1. **EC2 → Security Groups** → find `brain-sg` → Edit outbound rules → delete the `0.0.0.0/0` rule
2. Find `clinical-sg` → Edit outbound rules → delete the `0.0.0.0/0` rule
3. **VPC → Route Tables** → find `Hospital-Public-RT` → Subnet Associations → disassociate `Clinical-Zone`

> ⚠️ Option B will cause `terraform.tfstate` to drift from real AWS state. Run `terraform plan` afterward to review the diff.

## 🔐 Security & Compliance

| Control | Implementation |
|---|---|
| Network isolation | All assets in a private VPC; outbound blocked post-bootstrap |
| No active exploits | Validation uses passive TCP handshakes only (`connect_ex`) |
| AWS AUP compliance | Passive, internal-only posture; no external-facing honeypot |
| AWS Pen Test Policy | No DoS simulation; no active attack tooling |
| Credential management | IAM instance roles for S3 access; no hardcoded keys anywhere |
| Admin access | AWS Client VPN is the only entry point (SSH, Kibana) |
| Elasticsearch security | `xpack.security.enabled=false` for dev — **must be enabled with TLS before any production use** |
| Log shipping | Beats traffic sent to port 5044 on The Brain |
| Data persistence | Elasticsearch logs survive container restarts via `es_data` Docker volume |

## 🐳 ELK Stack Details

All three services run on an isolated Docker bridge network (`honeypot-net`) so they communicate by container name without VPC exposure.

| Container | Port | Role | Memory Cap |
|---|---|---|---|
| `elasticsearch` | 9200 | Log storage and indexing | 2GB (`ES_JAVA_OPTS=-Xms2g -Xmx2g`) |
| `logstash` | 5044 | Ingests Beats from Clinical Zone | 1GB (`LS_JAVA_OPTS=-Xms1g -Xmx1g`) |
| `kibana` | 5601 | Web dashboard (via VPN) | — |

The `brain_bootstrap.sh` script sets `vm.max_map_count=262144` automatically before launching Docker Compose — Elasticsearch will crash immediately without this.

**Logstash pipeline** (`logstash.conf`):

```conf
input {
  beats { port => 5044 }
}
output {
  elasticsearch { hosts => ["elasticsearch:9200"] }
}
```

## ⚠️ Known Limitations & TODOs

- [ ] **No `terraform.tfvars`** — region (`us-east-1`) is only configurable in `variables.tf`; a `.tfvars` file would improve portability
- [ ] **No remote state backend** — `terraform.tfstate` is local; for team use, configure an S3 + DynamoDB backend for locking
- [ ] **Windows PowerShell bootstrap** — the Winlogbeat install script targets PowerShell 3.0/4.0 compatibility on Windows Server 2012 R2 and requires hands-on testing; consider it provisional
- [ ] **Windows 7 unavailable on AWS** — Windows Server 2012 R2 is the closest available substitute
- [ ] **xpack security disabled** — `xpack.security.enabled=false` must be flipped to `true` with TLS configured before any production-like deployment
- [ ] **Validation script port coverage** — the script knocks all four ports against all three hosts; ports that are genuinely closed still generate log events, but the `[KNOCK]` output could be misleading to a first-time reader
